# SoundBrake 仕様書 v1

> ざっくり仕様書。内部処理は既存の `extract_and_normalize.sh` のロジックをそのまま踏襲し、
> SwiftUIのネイティブUIから ffmpeg プロセスを呼び出す構成とする。

---

## 1. 概要

mp4ファイルから音声を抽出し、複数ファイル間でラウドネス（音量感）をそろえて `.m4a` として
書き出すMac向けの軽量ユーティリティ。CLIスクリプトで実現していた処理を、GUIでドラッグ&ドロップ
するだけで使えるようにする。

- **対象OS**: macOS 14+（PicoShopと同じ想定ライン）
- **UI**: SwiftUI
- **内部処理**: システムにインストール済みの ffmpeg をサブプロセスとして呼び出す（再実装はしない）
- **想定ユーザー**: 自分（エンジニア）。ffmpegはHomebrewで導入済み前提でOK

---

## 2. 想定ワークフロー

1. Finderから mp4 ファイルを複数選択してウィンドウにドラッグ&ドロップ
2. 出力先フォルダを指定（フォルダピッカー）
3. 「変換開始」ボタンを押す
4. ファイルごとに「解析中→変換中→完了」の進捗がリストで見える
5. 途中で気が変わったら「中止」でいつでも止められる
6. 気が向いたら「再開」を押せば、待機中だったファイルだけ続きから処理される
7. 出力先フォルダに、元と同じファイル名で `.m4a` が並ぶ

---

## 3. 機能要件

### 3.1 ファイル入力
- ウィンドウ全体（またはドロップ専用エリア）に mp4 をドラッグ&ドロップで追加可能
- 「ファイルを選択...」ボタンからの複数選択にも対応（NSOpenPanel）
- フォルダごとドロップされた場合は中の mp4 を再帰的に拾う（任意・v1では優先度低）
- 追加済みファイルは、**現在ffmpegが処理中の行を除き、いつでもリストから削除できる**
  （待機中・完了・失敗・スキップ、どの状態でもOK。変換中の全体処理を止めなくても、
  まだ手をつけていない待機中ファイルをその場で間引ける）
- リストから削除しても、**既に書き出し済みの出力ファイル自体は削除しない**
  （あくまで一覧表示から外れるだけ。誤操作でディスク上のファイルまで消えないようにする）
- 処理中の行だけは削除ボタンをグレーアウトし、誤って中断させないようにする
- 対応拡張子は `.mp4` を主対象とし、内部的には ffmpeg が音声トラックを持つ他コンテナ
  （`.mov` など）も扱えるため、将来的な拡張は容易な設計にしておく

### 3.2 出力先指定
- フォルダピッカーで出力先を指定（NSOpenPanel、ディレクトリ選択モード）
- 前回指定したフォルダを記憶しておく（UserDefaults）
- 出力ファイル名は「元のファイル名（拡張子なし）+ .m4a」で統一。サフィックスは付けない
- 出力先に同名ファイルが既に存在する場合はスキップし、リスト上で「スキップ（既存）」と表示

### 3.3 変換処理（コア）
既存スクリプトと同じロジックをそのまま踏襲:

1. **1パス目（解析）**: `ffmpeg -i <入力> -vn -af loudnorm=I=-16:LRA=11:TP=-1.5:print_format=json -f null -`
   でラウドネスを測定し、JSON結果をパース
2. **2パス目（変換）**: 測定値を使って `loudnorm` に `measured_I` などを渡しつつ、
   `-vn -c:a aac -b:a 192k` で音声のみ抽出＆正規化して出力
3. 目標ラウドネス値（デフォルト -16 LUFS）は設定画面 or 詳細オプションで変更可能にする

> ネイティブアプリ化のメリット: シェルスクリプト版で発生していた
> 「`find | while read` パイプとffmpegのstdin競合」問題が構造的に起きない
> （`Process` はジョブごとに独立したstdin/stdoutを持てるため、`-nostdin` 相当の対策が自然に効く）

### 3.4 進捗表示
- ファイルごとに以下を表示:
  - ファイル名
  - 状態（待機中 / 解析中 / 変換中 / 完了 / 失敗 / スキップ）→ かわいいアイコン表示の詳細は4.2.1参照
  - ※中止されたジョブは「待機中」に戻る運用のため、独立した「キャンセル済み」状態は持たない（3.5参照）
  - 進捗バー（0-100%）
- 進捗は ffmpeg の stderr に出る `time=00:03:12.34` を、事前に取得した総尺と比較して算出
  - 1パス目・2パス目それぞれで進捗を取り、全体を「1パス目50% + 2パス目50%」で合算する
- リスト全体の上部 or 下部に「全体進捗（例: 3/8 完了）」のサマリを表示

### 3.5 中止機能（再開のしやすさを重視）
- 「中止」ボタンで、実行中のffmpegプロセスを `terminate()` し、以降のキュー処理を止める
- 中止時、処理中だった出力ファイル（未完成のもの）は削除する
- **処理中だったジョブのステータスは「待機中」に戻す**（2パス方式は途中から再開できないので、
  中断されたファイルは次回また最初からやり直す設計にする。中途半端な「キャンセル済み」状態を
  作らず、単純に「未処理」に戻すことで再開ロジックをシンプルに保つ）
- 既に完了しているファイル・失敗したファイルのステータスはそのまま保持する（やり直さない）
- 中止後、リストは「完了／失敗／待機中」が混在した状態になる。ここで
  **「変換開始」ボタンは、待機中のジョブが残っている場合「再開」に表示が変わる**
  （新規ファイルのみ、または全件完了後は「変換開始」のまま）
- 「再開」を押すと、待機中のジョブだけを対象に処理を続行する（完了済みは触らない）
- 出力先の「同名ファイルが存在すればスキップ」ロジックが安全装置としても働くので、
  中止→再開を何度繰り返しても、完了済みファイルを誤って壊すことはない
- リストの状態はアプリを閉じない限りセッション中保持される（アプリ再起動をまたいだ永続化はv1では対象外。
  必要になったら7章の今後の検討事項に追加する）

### 3.6 エラーハンドリング
- 1ファイルが失敗しても全体は止めず、次のファイルへ進む
- 失敗したファイルはリスト上に理由を短く表示（例: 「解析失敗」「デコードエラー」）
- 詳細なffmpegログは、行をクリック/ホバーで展開できると尚良い（v1では任意）

## 4. UI設計

### 4.1 画面構成（ワイヤーフレーム）

```plantuml
@startsalt
{
  {* ファイルをドラッグ&ドロップ、または + ボタンで追加
  }
  {#
  ファイル名                  | 状態              | 進捗              | 
  全力少年.mp4                | 変換中            | [====------] 42% | (削除不可)
  空も飛べるはず.mp4          | 完了              | [==========] 100%| [x]
  どんなときも。.mp4          | 待機中            | -                 | [x]
  only_my_railgun.mp4         | 失敗: デコードエラー | -              | [x]
  }
  --
  出力先  | "~/Music/正規化済み" | [選択...]
  目標ラウドネス | "-16 LUFS ▾"
  --
  [+ ファイル追加] | [変換開始] | [中止]
  --
  全体進捗: [======----] 3/8 完了
}
@endsalt
```

（いつものKrokiサーバーに投げればプレビューできるはず）

※「変換開始」ボタンは、中止後に待機中ジョブが残っていれば「再開」に変わる（4.2参照）

### 4.2 デザイン方針
- ミニマルでノイズの少ないUI。機能過多にしない（みゃあの好みに寄せる）
- 状態はアイコン＋色で直感的に（詳細は4.2.1）
- ドラッグ&ドロップ中はドロップ可能エリアをハイライト（枠線 or 背景変化）
- ダークモード対応必須（システム設定に追従）
- リストは `List` + カスタム行ビューで、ファイル名は長くても省略記号で1行に収める
  （フルパスはホバーでツールチップ表示）
- 変換中は「変換開始」ボタンを「中止」に差し替えるか、隣に中止ボタンを常設するかは
  実装時に決めてOK（個人的には常設の方が事故りにくいので常設推奨）
- メインボタンのラベルは状態に応じて動的に変化させる:
  「変換開始」（初回・全件完了後） ⇔ 「再開」（中止後、待機中ジョブが残っている場合）
- 各行に削除ボタン（×やゴミ箱アイコン）を用意。処理中の行だけ非活性（グレーアウト）にする

### 4.2.1 状態アイコン（かわいさ重視）

無機質なシステムっぽさを避けて、パステル背景＋丸っこいアイコンの「バッジ」で表現する。
ステータス列は素のSF Symbolsをポンと置くだけじゃなく、丸背景チップ（pill型）に収める。

| 状態     | SF Symbol候補                     | 背景色（パステル） | アイコン色 | 補足演出                         |
|----------|-----------------------------------|--------------------|------------|----------------------------------|
| 待機中   | `moon.zzz.fill`                   | ライトグレー        | グレー     | 控えめ、まだ寝てる感              |
| 解析中   | `waveform.and.magnifyingglass`    | ライトブルー         | ブルー     | ゆる〜くパルス（脈動）アニメ       |
| 変換中   | `waveform`                        | ライトブルー〜パープル | パープル   | 波形が小さくバウンス/揺れるアニメ  |
| 完了     | `checkmark.seal.fill`             | ライトグリーン       | グリーン   | 表示時にぽよんと拡縮するポップアニメ |
| 失敗     | `exclamationmark.triangle.fill`   | ライトオレンジ       | オレンジ   | 赤は避けて、責めてる感を出さない   |
| スキップ | `arrow.uturn.forward.circle.fill` | ライトグレー         | グレー     | サラッと控えめに                  |

- 「失敗」をあえて赤にせずオレンジ寄りにするのは、失敗＝悪いことという圧を弱めて
  「ちょっと待って、こっちだけ後で見てね」くらいのトーンに抑えるため
- 完了時のポップアニメ（`scaleEffect` + `spring` アニメーション）は地味に効くので、
  ここだけは手を抜かずに気持ちよさを作る
- 全体進捗バーの端にも、全部終わったらちょこっと跳ねる演出を入れると達成感が出て良さそう
- 絵文字（🌙 🔍 🎵 ✅ ⚠️ ↩️）で簡易的に代替する案もアリ。SF Symbolsより実装が速いので、
  MVP段階では絵文字、v1.1で正式なアイコン＋アニメに差し替える進め方でもOK

### 4.3 状態遷移（ファイル単位）
```
待機中 → 解析中 → 変換中 → 完了
              ↘        ↘ 失敗
               失敗
待機中 → スキップ（既存の出力ファイルがある場合）

[中止ボタン押下時]
解析中/変換中（実行中だった1件） → 待機中 に差し戻し（未完成の出力は削除）
待機中（まだ手つかずの残り） → そのまま待機中
完了/失敗（既に終わっていたもの） → 変更なし

[再開ボタン押下時]
待機中 のジョブから処理を再開（完了/失敗は触らない）
```

## 5. 技術構成

### 5.1 プラットフォーム/言語
- Swift + SwiftUI（AppKitへの部分的なフォールバックは必要に応じて: NSOpenPanel等）
- 非同期処理は `async/await` + `Task` で1ファイルずつ、または限定並列（後述）で処理

### 5.2 ffmpeg呼び出し方式
- `Process()` でffmpegバイナリを直接起動
- バイナリ探索順: `/opt/homebrew/bin/ffmpeg` → `/usr/local/bin/ffmpeg` → `PATH` 経由
- 見つからない場合はアラートで `brew install ffmpeg` を案内（v1はバンドルせず、
  システムのffmpegに依存する方針。将来的にアプリ内バンドルも検討可）
- 各プロセスの `standardInput` は明示的に `FileHandle.nullDevice` にしておく
  （念のためのstdin対策。ネイティブでは基本不要だが保険として）

### 5.3 処理パイプライン（1ファイルあたり）
```
1. 出力先に同名ファイルが存在するか確認 → あればスキップ
2. ffmpeg 1パス目起動（loudnorm 測定、-f null -）
   → stderr をリアルタイムでパースして進捗更新
   → 完了後、JSON部分を正規表現 or 簡易パーサで抽出
3. JSON取得失敗 → 「解析失敗」として記録、次のファイルへ
4. ffmpeg 2パス目起動（measured_I 等を渡して実変換）
   → stderr の time= を進捗に反映
5. 成功 → 「完了」、失敗 → 「失敗」として記録
```

### 5.4 並列度
- v1はシンプルに1ファイルずつ逐次処理でOK（元スクリプトと同じ挙動）
- 将来的にCPUコア数に応じて2-3並列にする余地はあるが、進捗UIが複雑になるので後回し推奨

## 6. データモデル（ざっくり）

```swift
enum ConversionStatus: Equatable {
    case queued
    case analyzing(progress: Double)
    case converting(progress: Double)
    case done
    case skipped(reason: String)
    case failed(reason: String)
    // 中止時は analyzing/converting → queued に差し戻すだけなので、
    // 専用の cancelled ケースは持たない（3.5参照）
}

struct ConversionJob: Identifiable {
    let id = UUID()
    let sourceURL: URL
    var outputURL: URL
    var status: ConversionStatus = .queued
}

// リストからの「削除」はジョブ配列から要素を取り除くだけの操作。
// 出力ファイルには一切触れない。status が .analyzing/.converting の行は
// UI側で削除ボタンを非活性にし、削除自体を呼ばせない。
```

## 7. 非対応・今後の検討事項（v1スコープ外）

- 抽出のみ／正規化のみ、と処理を分けて選べるトグル（今回は常に両方セットで実行）
- 目標ラウドネスのプリセット切り替え（配信用/放送用など）
- ffmpegバイナリのアプリ内バンドル（配布のしやすさ重視するなら将来対応）
- 変換後ファイルのプレビュー再生
- 複数フォルダのドロップ時の再帰探索

---

## 8. マイルストーン案

1. **MVP**: ドラッグ&ドロップ→出力先指定→変換→完了、のリニアな流れが動く（進捗は簡易でOK）
2. **v1**: 進捗バーの精緻化、中止機能、エラーハンドリングの作り込み
3. **v1.1**: UIの磨き込み（アニメーション、状態アイコンの質感など）
